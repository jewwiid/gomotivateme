## Goal
Let users post progress updates against specific milestones. Each milestone becomes a container for its own feed of notes/photos/links — showing how the user is actively working toward that milestone. This transforms milestones from a static 4-item checklist into a living progression log.

## The key insight
The data plumbing already exists: `updates.milestoneId` field is in the schema, and `updates.add` already accepts it as an argument. It's just never used by any caller. So this is primarily a UI + query feature, not a schema migration.

## Part 1 — Schema: add a milestone-scoped index

### `convex/schema.ts`
Add `.index("by_goal_milestone_created", ["goalId", "milestoneId", "createdAt"])` to the `updates` table. This lets us efficiently query "all updates for milestone X" without a full scan. Existing indexes stay — the new one is additive.

## Part 2 — Backend: milestone-filtered query + validation

### `convex/updates.ts` — `listForMilestone` query
New query: `{ goalId, milestoneId }`. Returns updates where `milestoneId` matches, newest first, public-visible only. Uses the new `by_goal_milestone_created` index. Visible to anyone (public updates on a public goal).

### `convex/updates.ts` — validate `milestoneId` in `add`
The `add` mutation already accepts `milestoneId` but doesn't validate it. Add a check: if `milestoneId` is provided, fetch the goal and verify that milestone id exists in `goal.milestones`. Prevents orphan/dangling references.

## Part 3 — UI: expandable milestones with per-milestone update feeds

### `components/MilestonesList.tsx` — restructure
Transform the flat checklist into expandable milestone cards. Each milestone becomes:
- **Collapsed state**: checkbox + title + a small count badge showing how many updates are posted against it ("3 updates")
- **Expanded state**: checkbox + title + a vertical feed of milestone-specific updates + an inline composer to add a new note

When expanded:
- Shows `listForMilestone` results — notes/photos/links posted against this milestone, rendered similarly to timeline entries (date, type icon, note text)
- Shows a compact "Add update" textarea at the bottom (owner only) that calls `updates.add` with `milestoneId` set
- The milestone completion entry (from `toggleMilestone`) also shows in this feed since it has `milestoneId` set

Expand/collapse toggles via clicking the milestone row (not the checkbox — checkbox stays separate for toggling done/undone). Use local state per-milestone for expansion.

### Owner vs visitor
- **Owner**: sees expand button, can post updates against milestones, can toggle done/undone
- **Visitor**: sees expand button (to read the progression), can see per-update reactions (already built), but no composer. The "Add update" textarea is owner-only.

## Part 4 — Timeline integration (light touch)

### `components/EditorialTimeline.tsx`
When rendering an update that has `milestoneId`, show a small badge/pill indicating which milestone it's associated with (e.g. "Research phase"). This gives context in the main timeline without requiring the user to expand the milestone section. The `UpdateDoc` interface gets an optional `milestoneId` field; I'll pass the milestones array to resolve the title.

## Out of scope
- Editing/reordering milestones
- Custom milestones (currently category-defaulted at creation)
- Milestone-level progress bars or percentages
- Moving the composer to support photos/links per milestone (starts with notes; can add media later)
