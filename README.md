# myodyssey

A public motivation app. Set a goal, share a single link, and let the people cheering you on drop thumbs-ups and notes as you work toward your target date.

This is the **MVP** — focused on the core loop:

1. Sign up (email + password).
2. Create a goal in a category (weight, fitness, learning, habit, creative, business, or custom).
3. Set a start value, target value, unit, and a target date.
4. Get a public link like `https://myodyssey.app/o/lose-20kg-by-summer-a1b2c`.
5. Share that link. Anyone can visit it.
6. Visitors see a vertical timeline of your progress and can:
   - Tap a thumbs-up (anonymous, one per visitor).
   - Leave you a message (with optional display name).
7. You log updates: notes, images, links, or a new measured value. Each update lands on the public timeline.
8. You decide which messages become public on the timeline (default: pending approval).

## Tech stack

- **Next.js 15** (App Router) hosted on **Vercel**.
- **Convex** for the database, real-time sync, file storage, and auth.
- **Convex Auth** (`@convex-dev/auth`) for email/password sign-in.
- **Tailwind CSS v4** for styling.
- **Framer Motion** for animations (progress bar, badge unlocks, reactions).
- **TypeScript** end to end.

## Local dev (first time)

```bash
# 1. Install
npm install

# 2. Spin up Convex — this will create a project + dev deployment,
#    generate the type bindings into convex/_generated/, and write
#    NEXT_PUBLIC_CONVEX_URL to .env.local
npx convex dev

# 3. In a second terminal, start Next.js
npm run dev
```

Open http://localhost:3000. The first sign-up is the owner of the workspace.

> **Why does `convex dev` need to run first?**
> The TypeScript bindings in `convex/_generated/` are generated by Convex. The repo ships with stub bindings so the project typechecks before you've set up a Convex project — running `npx convex dev` once replaces them with the real, schema-aware types.

## Deployment to Vercel

1. Push this repo to GitHub.
2. In Vercel, import the repo.
3. Set environment variables in the Vercel project settings:
   - `NEXT_PUBLIC_CONVEX_URL` — the production Convex deployment URL (`npx convex deploy` prints it).
   - `NEXT_PUBLIC_SITE_URL` — your Vercel deployment URL (e.g. `https://myodyssey.app`).
4. Run `npx convex deploy` once locally to ship the schema and functions to the production deployment.
5. Deploy from Vercel. Done.

## Notes for the developer

- The convex functions live next to the schema and use the `mutation` / `query` builders. The `_generated` folder is a stub before first `convex dev` run; real codegen replaces it.
- `getAuthUserId` from `@convex-dev/auth/server` is the server-side auth check; `useConvexAuth` from `convex/react` is the client-side auth state hook.
- The "support url" in the spec maps to `NEXT_PUBLIC_SITE_URL` — used to build the absolute share link shown in the dashboard.
